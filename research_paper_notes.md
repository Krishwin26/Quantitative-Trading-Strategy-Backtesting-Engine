# Research Paper Notes: Database-Driven Quantitative Trading Strategy Engine

The following document synthesizes technical notes intended to support a research or project paper discussing the integration of RDBMS with Quantitative Finance methodologies.

## 1. System Architecture

The Quantitative Trading Engine architecture is delineated into three core interconnected layers: Application, Analytical, and Data Storage.
- **Application Layer**: Responsible for managing data pipelines (e.g., CSV imports, API calls), backtest loop logic, and daily order execution simulation (`data_loader.py`, `backtester.py`, `main.py`).
- **Analytical Layer**: Implements purely functional quantitative methodologies containing no state. The strategy logic acts purely on input DataFrames transforming them into signals (Z-Score calculation). Performance evaluation occurs independently by deriving statistical significance from resulting data stores (`strategy.py`, `metrics.py`).
- **Data Storage Layer**: Functions as the definitive source of truth providing ACID (Atomicity, Consistency, Isolation, Durability) guarantees, crucial when simulating multi-year financial ledgers. Powered by an Object-Relational Mapper (ORM), the system operates agnostically across SQLite, PostgreSQL, and MySQL.

## 2. Database Schema

The relational database model ensures the elimination of data redundancy and strict temporal consistency through properly defined keys and indexes.

- **`Stocks` (Entity)**: Central registry of tradable assets (`stock_id`, `ticker`).
- **`Price_Data` (Time-Series)**: Standardizes historical OHLCV data. A composite `UNIQUE` index on `(stock_id, date)` acts as a constraint against data duplication ensuring temporal integrity.
- **`Trading_Signals` (Fact)**: Records actionable insights generated by the model. 
- **`Trades` (Transaction Ledger)**: Persists completed round-trip executions including calculated PnL. The relational link ensures performance can be analyzed on a per-asset basis.
- **`Portfolio_Performance` (State Snapshot)**: Logs daily aggregate portfolio evaluations essential for calculating risk-adjusted metrics (Maximum Drawdown, Volatility).

*Architectural Justification: Storing transactional signals and portfolio states allows for complex post-execution SQL aggregations (e.g., win rates grouped by market sector) reducing memory overhead typically necessary in entirely Pandas-based pipelines.*

## 3. Algorithm Workflow

The algorithm follows an end-to-end Extract-Transform-Load-Execute (ETLE) cycle:
1. **Extraction**: Historical market data is sourced (or synthetically generated) and ingested into memory via Pandas.
2. **First Load**: Raw pricing data is stored sequentially inside the robust SQL `Price_Data` table.
3. **Transformation**: Time series data is extracted back out of the DB to evaluate the **Mean Reversion logic**. A 20-day Simple Moving Average (SMA) and Standard Deviation are calculated to derive a normalized Z-Score. 
4. **Second Load**: The generated 'BUY', 'SELL', or 'HOLD' decisions are inserted into `Trading_Signals`.
5. **Execution Loop**: The Backtester iteratively steps through the historical timeline querying both pricing and signals. Positions are opened allocating 50% of available cash on 'BUY' condition (Z-Score < -2.0) and liquidated on 'SELL' condition (Z-Score > 2.0). 
6. **Final Load**: Complete trade statistics and daily net liquidating values are committed back to the database.

## 4. Experimental Setup

The experiment evaluates the efficacy of a 20-day Mean Reversion strategy operating under a simulated, frictionless backtest environment.

- **Dataset**: Synthetically generated 1,000-business-day market data mimicking standard Equity returns utilizing a Geometric Brownian Motion model. Two securities (simulating TSLA and AAPL volatility profiles) were primarily evaluated.
- **Capital Constraints**: The portfolio was instantiated with an initial cash value of $100,000.00.
- **Assumptions**: The executing system assumed ideal liquidity allowing executions strictly at the daily 'Close' price upon signal generation. Slippage and brokerage commissions were assumed zero for the baseline performance readouts.
- **Metrics Evaluated**:
    - **Annualized Return**: Average yearly compound return.
    - **Volatility**: Annualized standard deviation of daily portfolio returns.
    - **Sharpe Ratio**: Risk-adjusted excess return over a static 2.0% Risk-Free Rate.
    - **Maximum Drawdown**: Worst-case historical peak-to-trough depreciation measuring absolute catastrophic risk.
